timeConfidence covers the full epistemic scale — including "rectified" for times derived by an astrologer from life events, which has no equivalent in the simpler field.  birthTimeTrust is a three-value signal (exact / approximate / unknown) suited for apps that don't need to distinguish rectification from other approximations.  Both fields are optional; use one or both — they're complementary, not redundant.
privacy & encryption
visibility meaning
"public" Anyone on ATProto can read the record.
"private" Convention — DID owner only. Not enforced by the protocol. Encrypt birthDatetime as JWE for real confidentiality.
"friends" Reserved. Social graph access — future feature.
For visibility: "private" — encrypt birthDatetime as JWE via jose and store ciphertext in encryptedData. Omit the plaintext field. 
// encrypt birthDatetime before posting a private chart
import { CompactEncrypt } from 'jose';

const jwe = await new CompactEncrypt(new TextEncoder().encode(birthDatetime))
  .setProtectedHeader({ alg: 'RSA-OAEP-256', enc: 'A256GCM' })
  .encrypt(recipientPublicKey);

const record = {
  encryptedData: jwe,
  visibility:    "private",
  // birthDatetime omitted
};

Data retention

Records exist only in your PDS. Delete anytime via com.atproto.repo.deleteRecord. No central server retains copies. AppViews may cache — ATProto's firehose makes erasure eventual, not instant.

Compliance

You control your PDS; you are the data controller. subjectDid creates identity linkage — set only with consent. GDPR right to erasure: delete the record. Age-gating for minors is an app responsibility.

verification · space.chartid.chartVerification

Rectification

When birth time is unknown, a skilled astrologer derives it from life events. verificationLabel: "astrologer-attested" records who did that work and makes it citable by any app.

Witness

A family member, physician, or registrar who knows the birth time can post an attestation from their own DID. Their key signs it — accountability is cryptographic, not social.

Self-declaration

"self-declared" is a valid label. Not all charts need external verification to be useful. Apps decide how much weight to assign each label type.

verificationLabel meaning
"birth-certificate-verified" Verifier has seen an official birth record. Strongest signal.
"astrologer-attested" Professional astrologer vouches for chart accuracy — date, time, place, and calculation method.
"historical-figure" Publicly documented time and place for a historical or public entity. No private birth record required.
"self-declared" Subject attests to their own data. No external verification.

Verifier graph

Each attestation lives in the verifier's own repo, signed by their key. Any AppView can query all chartVerification records pointing to a given chartUri. The union is the trust graph. Weight by social graph, attestation count, or verifier reputation. No central authority required.

Composable by design

Multiple verifiers can independently attest to the same chart. A feed generator showing "verified charts only" queries the firehose for space.chartid.chartVerification records and filters by label. No special index or permissioned API needed.

settings · space.chartid.settings

The settings record is a singleton — key: "self" means exactly one can exist per DID, written and overwritten in place rather than appended to a collection. Apps read it to resolve a user's primary chart without guessing which of many natalChart records to surface: western, jyotish, rectified versions all coexist; primaryChartUri is the explicit pointer. The same record carries visibility defaults and tradition context so apps can set sensible UI defaults without asking the user to configure them twice.

Self-sovereign

Records live in the user's AT Protocol repo. Signed by their key. Portable across every PDS. Deletable by them at any time.

Multi-tradition

traditions: ["western", "jyotish"] — one record, both systems. Apps ignore traditions they don't implement. No forking required.

Partial charts valid

Unknown birth time? timeConfidence: "unknown". Planets without houses are a valid, useful record. The schema doesn't require impossible data.

Open standard

No app owns space.chartid.*. Steward domain: chartid.space. RFC open for comment. CC0 — use freely.

contributing

Schema changes follow semantic versioning — patch for description improvements, minor for new optional fields, major for breaking changes requiring a new lexicon file and RFC comment period. All proposals must pass the lexicon style guide (no floats, knownValues not closed enums, explicit house systems) and a privacy review checklist for any field that touches PII.
